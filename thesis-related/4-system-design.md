# 4. System Design

## 4.1 Pipeline Overview

The system is organized into four processing layers, each with a distinct responsibility. Layer 1 (Capture) operates on the iOS device, collecting raw sensor data through the camera, microphone, and accelerometer. Layer 1.5 (Preprocessing) runs on the same device, cutting the frame stream into scene-level batches, filtering low-quality content, and selecting a compact set of representative key frames from each scene. Layer 2 (Inference) runs on a local server, processing each batch through a vision-language model to produce structured text descriptions of physical-world context. Layer 3 (Memory Integration) writes those descriptions into OpenClaw's markdown memory architecture at three levels of temporal granularity.

\[Figure 1: Pipeline architecture. Layer 1 captures frames at adaptive intervals triggered by audio and motion events; Layer 1.5 cuts scene batches, filters frames, and selects key frames; Layer 2 runs VLM inference per batch; Layer 3 writes to physical-logs, physical-insights, and physical-pattern.md.\]

The four layers are connected by two transfer mechanisms. The iOS device sends processed key-frame batches to the local server over HTTP; the server writes VLM outputs directly into OpenClaw's memory directory. This design keeps inference off-device—preserving battery and compute budget—while keeping memory writes local and human-readable.

## 4.2 Capture Layer

**Sensor roles.** The capture layer uses three sensors: the rear camera, the microphone, and the accelerometer. Each sensor serves a dual role—triggering adaptive interval adjustments, and contributing metadata attached to each captured frame. These roles are distinct: trigger behavior controls *when* frames are captured; data behavior controls *what* information accompanies them.

**Vision.** The camera operates in adaptive interval mode, capturing frames at intervals that vary between minInterval and maxInterval. At rest—when no triggers have fired—the system captures at maxInterval, a rate sufficient to detect slow-moving scene changes without generating redundant frames between them. When a trigger fires, the interval drops immediately to minInterval, ensuring the pipeline captures dense coverage during activity transitions. After the trigger subsides, the interval ramps back toward maxInterval geometrically, governed by rampRatio. This ramp prevents the system from flooding the batch with near-identical frames once a brief event ends.

The use of adaptive rather than fixed-rate capture is a deliberate design choice. Fixed high-rate capture wastes storage and tokens on static scenes; fixed low-rate capture misses brief events. The sensor-trigger architecture resolves this by coupling capture density to event detection: the camera becomes attentive precisely when something changes.

**Audio.** The microphone plays both trigger and data roles. As a trigger, voice activity detection (VAD) monitors the audio stream continuously; speech onset and offset each fire a trigger, dropping the capture interval to minInterval. This ensures that conversations—which often co-occur with changes in social context or activity—are densely covered. As a data source, two tags are attached to each frame: a VAD tag indicating whether speech was active at capture time, and a noise level tag classifying the ambient sound environment. Speaker count and speaker identity are excluded: the pipeline assigns VLMs responsibility for social context inference from the visual frame, avoiding the additional privacy exposure that diarization would require.

Audio transcription (transcriptionEnabled) is configurable but disabled by default. When enabled, on-device speech recognition produces a transcript for each speech segment, attached to frames captured during that segment. The transcription flag is treated as a study variable: it introduces richer linguistic content but adds on-device compute load and raises additional privacy considerations.

**IMU.** The accelerometer serves both trigger and data roles. A rolling variance computation over a short window classifies motion as stationary or sustained_motion. State transitions—stationary to sustained_motion and back—fire triggers, dropping the capture interval to minInterval. As a data source, the current motion state is attached to each frame as a binary tag. More granular activity classification is excluded: phone wearing posture varies enough across contexts that accelerometer-based activity classifiers are unreliable without per-user calibration. The VLM infers specific activity from the visual frame, where posture, surroundings, and objects provide more reliable evidence than accelerometer signals alone. The threshold sustainedMotionThreshold governs how long high-variance motion must persist before the trigger fires, preventing brief phone adjustments from triggering unnecessarily.

## 4.3 Preprocessing

**Why preprocess.** Token cost and information density are both arguments for preprocessing. Submitting a raw frame sequence to a vision-language model would include many near-identical frames and many low-quality frames—blurry, dark, or visually uninformative. These frames consume tokens without contributing to the VLM's understanding of the scene. The preprocessing layer resolves this by identifying scene boundaries, filtering poor-quality frames, and selecting a compact, information-dense set of key frames from each batch. The goal is to maximize the signal the VLM receives per token spent.

**Batch boundary detection.** Batches are delimited by a two-stage process. Stage 1: a sensor trigger (VAD onset/offset or IMU motion state change) proposes a cut. Stage 2: visual similarity verification—comparing the incoming frame against the last frame in the current buffer, or the last surviving frame of the previous batch when the buffer is empty—confirms or rejects the proposal. If similarity falls below ssimBoundaryThreshold, the scene has genuinely changed and the batch is confirmed; if similarity remains high, the sensor event likely reflected a local fluctuation and capture continues. This two-stage design handles a fundamental ambiguity: sensor events do not reliably track scene changes, and visual similarity alone, without a trigger, would rely entirely on time-based cutting. The combination allows sensor events to propose boundaries while visual content arbitrates.

Two time-based fallbacks force cuts regardless of sensor state. For the first batch of a session, a cut fires after firstBatchWindowSeconds—a shorter window, since the first batch has no prior tail for comparison. Subsequent batches cut after maxWindowSeconds. These fallbacks ensure the pipeline produces output even during long stable scenes. Both thresholds are experiment variables resolved during Phase 0.

**Frame filtering.** Within each confirmed batch, frames pass through a sequence of filters before scoring. Black frames—those with extremely low mean pixel brightness—are removed; they typically reflect lens obstruction or capture during transitions and carry no visual information. Blurry frames—those whose Laplacian variance falls below a sharpness threshold—are removed; blur indicates camera motion or poor focus and degrades VLM input quality. Frames that survive filtering inherit the trigger events of any deleted frames that immediately preceded them: if a VAD onset occurred on a frame subsequently removed as blurry, the next surviving frame carries that event forward. This inheritance ensures that trigger information is not silently discarded when the frame that generated it fails quality filtering.

**Importance scoring.** Surviving frames are scored across four dimensions: visual quality (sharpness), audio state, IMU state, and temporal sparsity (time elapsed since the last selected frame, rewarding temporal spread across the batch). The audio and IMU dimensions are change-aware: onset and offset events receive higher scores than sustained states, on the grounds that transitions carry more information for a VLM reconstructing what happened in a scene than does stable continuation. Inherited trigger events—carried forward from deleted frames—receive elevated scores as well, since they indicate that something changed at that point in the scene even if the frame that captured it was filtered out. The four dimension weights—wVisual, wAudio, wIMU, wSparsity—are experiment variables resolved during Phase 0.

**Deduplication and key frame selection.** Scored frames are sorted by composite score and passed through greedy deduplication: iterating from highest to lowest, a frame is dropped if its visual similarity to any already-kept frame exceeds ssimDedupThreshold. This ensures the surviving set is both high-quality and non-redundant. The target number of frames is computed dynamically as K = clamp(⌈duration_min × kDensityPerMin⌉, kMin, kMax), scaling with batch duration so that longer scenes receive proportionally more coverage without exceeding kMax. Frames whose score meets or exceeds scoreThreshold are guaranteed inclusion regardless of K; remaining slots up to K are filled by the next-best scoring frames. If no frames survive filtering entirely, the batch is discarded.

## 4.4 Inference

**Path A versus Path B.** The most fundamental inference design question is the granularity of VLM calls: per-frame inference (Path A) or per-batch inference (Path B). Path A submits each captured frame individually, producing a description per frame. Path B submits a scene-level batch—multiple key frames with shared context—and produces a single description covering the scene.

Per-frame inference is rejected on three grounds. First, adjacent frames are visually similar enough that per-frame descriptions will be largely redundant; the marginal information per token is low. Second, per-frame inference precludes cross-frame reasoning: understanding what changed between the start of a scene and its end requires a model that sees both. Third, API cost scales with frame count rather than scene count, making sustained deployment expensive without a corresponding gain in personalization-relevant information. Per-batch inference treats each scene as the unit of analysis, enabling the VLM to reason about transitions, sequences, and context across the batch duration.

**VLM input construction.** Each inference call consists of three components. A preamble provides batch-level context: the capture timestamp, batch duration, total frames captured versus key frames submitted, and the VLM summary of the prior batch from the same session if one exists. This cross-batch context—a single prior-batch summary stored per session—provides continuity across consecutive scenes without requiring the VLM to hold the full history in context. Per-frame annotations label each image with its index, relative timestamp within the batch, and sensor tags (motion state, speech detection, noise level); these ground the VLM's interpretation in the sensor evidence the pipeline has already classified. The key frames follow in chronological order.

**Output format.** The VLM is instructed to return a structured JSON object with six fields: activity, location, objects, social_context, notable_events, and observation. The first five are concise extractions; the observation field is a short narrative paragraph of four to six sentences in past tense and third person. This format serves two purposes: structured fields enable downstream processing and pattern analysis; the observation paragraph provides a human-readable summary suitable for direct insertion into the physical log. Fields for which the VLM finds insufficient evidence are set to "not observed" rather than hallucinated. The JSON is validated on receipt; malformed responses trigger up to two retries before the batch is logged with an error. The prompt wording that produces this output is treated as an experiment variable iterated during Phase 0 and frozen at Phase 1 start.

## 4.5 Memory Integration

**Three-tier direct write strategy.** VLM outputs are written directly into OpenClaw's memory directory at three levels of temporal granularity, rather than passing through OpenClaw's native memory ingestion mechanism. Direct write is chosen because OpenClaw's ingestion pipeline applies relevance heuristics designed for conversational memory; physical-world observations require different retention logic—every batch output should be stored, not filtered by conversational relevance. The three tiers serve distinct purposes.

physical-logs stores real-time batch outputs: each VLM response is appended immediately after inference, timestamped and keyed by batch and session identifiers. This tier is the complete, unfiltered record of what the pipeline made visible—the full-volume source that feeds higher tiers and supports on-demand retrieval.

physical-insights stores a rolling intra-day summary for the current date. An update is triggered automatically when insight_min_batches new batches have accumulated since the last update, or insight_min_minutes have elapsed—whichever fires first. When triggered, the VLM receives the new log entries since the last update alongside the current insight file, and produces an updated summary that incorporates both. Human annotations submitted via the web monitor are consumed at each insight update: they are appended to the VLM prompt with elevated priority and cleared from the pending queue after processing. This rolling mechanism ensures the insight file reflects the current state of the day without requiring a full re-read of the complete log on each pass.

physical-pattern.md stores a persistent behavioral profile updated nightly at nightly_hour. Each night, the VLM receives the current pattern file alongside the day's insight summary—and a partial summary from the current day if the update runs in the early morning—and produces a revised profile that merges new evidence into existing entries rather than appending. The pattern file is intentionally compact: it is loaded into OpenClaw's system prompt on every agent turn via a bootstrap hook, so its size directly affects the context budget available to the agent.

**Three-tier visibility mapping.** The three tiers are surfaced to the agent through distinct retrieval mechanisms matched to their access patterns. physical-pattern.md is injected at session bootstrap on every turn; it captures stable behavioral patterns that should always be available without requiring retrieval. physical-insights for the current and previous day are loaded at session start via an AGENTS.md instruction; they provide recent-day context without competing for the retrieval budget. physical-logs are indexed for on-demand retrieval; they are too voluminous for constant injection but searchable when a specific past observation is relevant to a task.

This visibility mapping reflects a deliberate trade-off between context budget and recency. The pattern file earns its constant presence by being compact and stable; the insight files earn session-start loading by being bounded to two days; the raw logs earn only on-demand access because their volume would exhaust the context budget if injected routinely.

**Retention.** Physical logs and raw frame files are retained for 14 days and then deleted. This window is long enough to capture weekly patterns—the most common recurrence period for behavioral regularities—while keeping storage bounded and limiting the scope of any privacy exposure. Distilled outputs (insight files, the pattern file) are retained indefinitely, as they contain no raw frames and represent already-abstracted summaries.

## 4.6 Implementation

The capture application is built for iOS using Swift, AVFoundation for camera and audio access, CoreMotion for accelerometer data, and Apple's Vision framework for frame similarity computation. The server is a Python Flask application that receives batches over HTTP, submits them to a vision-language model API—configurable as either Anthropic or OpenRouter via the provider and model parameters—and writes outputs to the file system. The server also hosts a browser-based web monitor for viewing recent batches, reviewing the three-tier memory output, and adjusting pipeline parameters without restarting the system.

The hardware platform is an iPhone carried on the body during waking hours—in a pocket or lanyard mount depending on the activity context. This is an acknowledged approximation of dedicated egocentric wearables such as smart glasses; the justification for this proxy is given in Chapter 3. The camera's field of view and mounting position vary more than they would with a purpose-built device, introducing additional noise into the visual record. This noise is treated as a study condition rather than a system failure: it reflects the kind of imperfect capture that any body-worn consumer device would produce in practice.

The study timeline maps onto the two phases described in Section 5.2. Phase 0 calibration runs for approximately one week, during which pipeline parameters are actively iterated and the design iteration log is maintained. At the end of Phase 0, all parameters are frozen. Phase 1 observation runs for two weeks with the system in continuous deployment during waking hours.
